# ADR-066: Autonomous Merge Pipeline — Gate Wall as Reviewer

**Status**: Proposed
**Date**: 2026-06-23 (revised; original 2026-06-22)
**Supersedes**: ADR-066 draft dated 2026-06-22 (two-lane model with CODEOWNERS hold)
**Relates to**: CI workflow (`.github/workflows/ci.yml`, `.github/workflows/release.yml`),
`scripts/apply-autonomous-merge.sh`, repository ruleset 17362266

---

## Context

khive receives a high and growing volume of pull requests, most generated by automated agents
working from issues. The prior draft of this ADR addressed that volume with a two-lane model: a
deterministic gate set for conforming changes, plus CODEOWNERS-enforced human review for
architecturally-significant paths. That model retains a human in the critical path for every
Lane-2 PR, which remains a throughput limit as automation intensity increases.

The maintainer's 2026-06-23 directive changes the trust model: the human gate moves from
per-PR to per-release. Merging to `main` is fully autonomous. Publishing a release version
remains human-gated. Safety is therefore delivered entirely by the deterministic CI gate wall,
which replaces what a human reviewer would otherwise catch on each diff. The gate wall must be
thick enough to justify that trust transfer.

The repository already enforces a substantial deterministic floor: format, clippy under
`-D warnings`, the full workspace test suite, a no-default-features build, a release build, SQL
lint, contract and smoke tests, deno tests, a JSON data-leak guard, a gitleaks secret scan, a
docs-format check, and a marketplace-example validator. Additional gates covering supply-chain
hygiene, API-surface stability, documentation build correctness, dependency review, and coverage
regression are being added by the companion implementation PRs (#215 and the new-gates PR).

## Decision

Adopt a single-lane autonomous merge pipeline.

Every PR that passes the full required gate wall (section 1) merges automatically to `main`
without human review. No pull request requires a human approver before merging to `main`.

The human gate is the release: no merge to `main` publishes anything. The npm artifact release
is held by a GitHub Environment protection rule (section 3); the crates.io library publish is a
separate maintainer-operated `make publish` step, run by hand from a machine that holds the
crates.io token. Neither publish path is reachable from autonomous trunk delivery. A merge to
`main` is not a release; it is the unit of continuous delivery into the trunk, which the release
gate samples when a human decides to publish.

A non-automatable floor (section 4) is never delegated to automation.

## Mechanism

### 1. Required gate wall (required status checks)

Branch protection on `main` (ruleset 17362266) requires all of the following status contexts.
Context strings are matched by exact name; the canonical list lives in
`scripts/apply-autonomous-merge.sh`, which is the source of truth for the exact strings and
must be kept in sync with this ADR.

Each gate in the wall replaces a category of defect that a human reviewer would otherwise be
responsible for catching.

**Correctness and code quality (replaces line-by-line review for mechanical errors)**

- `CI (ubuntu-latest)` and `CI (macos-latest)` — the matrix job running `scripts/ci.sh`: format
  check, clippy `--workspace --all-targets -D warnings`, workspace test suite,
  no-default-features check, release build, SQL lint, the no-stub guard, and
  contract/smoke/deno tests. Both matrix legs are required; a failure in either blocks merge.

**Supply-chain and security (replaces manual dependency and secret auditing)**

- `Supply-chain (cargo-deny)` — CVE advisory check, license policy, and bans policy against
  `crates/deny.toml`. Blocks any dependency with a CVSS severity at or above the configured
  threshold, any dependency under a disallowed license, and any banned crate.
- `JSON/JSONL data-leak guard` — detects committed test fixtures or serialized corpus files
  that must not enter the repository.
- `Secret scan (gitleaks)` — scans full commit history for credential patterns; a leaked secret
  is not correctable by revert because history persists.
- `Dependency review` — flags newly introduced dependencies with known vulnerabilities or
  license incompatibilities (GitHub native action, `pull_request` trigger).

**API stability and documentation (replaces review for interface breakage and doc rot)**

- `Doc build (-D warnings)` — runs `cargo doc --workspace --no-deps` with `-D warnings` so
  broken intra-doc links and malformed doc comments fail the gate rather than silently rot.

API-surface stability (SemVer) is deliberately **not** a per-PR required check. `cargo-semver-checks`
compares each crate against its crates.io baseline _at the same version_, so on a workspace that pins
a fixed dev version mid-cycle and bumps only at release it is permanently red — it can never go green
as a per-PR gate and would wedge the wall. SemVer is enforced at the boundary where the version
actually bumps and the check is green-able: the release gate (section 3).

**Quality ratchet (replaces review for regression in coverage and code hygiene)**

- `Coverage ratchet` — measures line coverage on the PR branch and blocks merge if coverage
  falls below the current `main` baseline. The baseline is the last recorded value on `main`; it
  can only be raised by a PR that adds tests, never lowered by a PR that removes them.
- `Docs lint` — enforces documentation formatting conventions.
- `Marketplace example validator` — validates the marketplace plugin examples remain
  syntactically correct.

**Conditionally-run regression gates (not in the required set — see below)**

- `Pipeline Regression Gate` (`bench-pipeline.yml`) and `ANN structural regression gate
  (synthetic, hermetic)` (`bench-1m.yml`) run only on PRs that touch specific retrieval and
  storage paths. A required status context that is never reported (because the workflow was
  skipped) permanently blocks merge for every PR on an unaffected path. These gates therefore
  cannot be globally required without wedging the majority of PRs. They are informational on PRs
  where they run. The `ann-ci-gate` job re-runs post-merge on every push to `main` as defense
  in depth.

The full, authoritative context list and the exact strings used in the ruleset are maintained in
`scripts/apply-autonomous-merge.sh`. When a new gate is introduced and its context name is
established, that script is updated first, and this ADR's prose is updated to match.

### 2. Auto-merge mechanics

Auto-merge is permitted repository-wide (`allow_auto_merge`). Squash is the merge method, so
each merge produces one revertable commit on `main`. Delete-branch-on-merge is enabled.

A lightweight workflow (`automerge.yml`) enables native auto-merge (`gh pr merge --auto
--squash`) on every PR when it becomes ready for review (the `ready_for_review` event; draft
PRs cannot carry auto-merge). This workflow uses a fine-grained token minted from a GitHub App
(a GitHub App bot) via a SHA-pinned token-minter action and runs on `pull_request_target` so it
has no access to the PR head's code. Its sole action is `gh pr merge --auto --squash`.

The bot proposer is an optional convenience layer, not a trust boundary. Its function is to
issue the `--auto` command so no human has to type it; the gate wall enforces what actually
merges. A PR opened by a human, by a different bot, or by any other means is treated identically
by the gate wall. There is no "proposer identity" check in the merge path.

Branch protection also enables "Require branches to be up to date before merging" and "Dismiss
stale pull request approvals when new commits are pushed." Because required approvals are set to
zero in this model, the dismissal rule has no operational effect; it is retained as a policy
invariant in case a future amendment re-introduces review requirements for a subset of changes.

One GitHub constraint governs the auto-merge workflow: the default `GITHUB_TOKEN` can enable
auto-merge, but events it triggers do not start new workflow runs. The auto-merge workflow only
arms the merge flag; it does not push commits or re-trigger CI.

A same-repository `pull_request` workflow runs the PR head's own workflow and script files, so a
PR that modifies `scripts/ci.sh` or a workflow file could weaken the gate that runs on itself.
The gate-defining path routing in section 3 documents this risk and the structural mitigation.

### 3. Release gate (human approval on publish)

Merging to `main` is not a release. Publishing is a separate step, gated on human approval.

`release.yml` triggers only on `workflow_dispatch` (the `v*` tag-push trigger was removed in
#222). It builds platform binaries and publishes the npm packages; the crates.io library publish is the separate
`scripts/publish.sh` (`make publish`) path. The npm publish job (`publish-all`) runs in a GitHub
**Environment** named `publish`. That environment carries a **required reviewer** protection
rule listing the maintainer. GitHub pauses the workflow at every job targeting `publish` and
waits for an authorized reviewer to approve the deployment before any publish step executes.

This means: any dispatch that reaches the publish step does so only after a human explicitly
approves the deployment in the GitHub UI. The environment gate is enforced by GitHub natively,
not by workflow logic. Even if `release.yml` is modified in a PR, the modification does not take
effect until the PR merges to `main` and a maintainer dispatches a release from `main`, at which
point the environment gate intercepts the publish job.

The npm token is scoped to the `publish` environment's secrets and is not accessible to any job
that does not run in that environment. This is the authorization boundary between autonomous
trunk delivery and npm publication. The crates.io library publish does not run in CI at all:
`scripts/publish.sh` (`make publish`) runs on a maintainer's machine using a local `cargo login`
token, so reaching crates.io already requires a human with publish rights invoking the command by
hand — no CI path, autonomous or otherwise, holds a crates.io token.

`workflow_dispatch` from `main` is the sole publish trigger. The workflow refuses a dispatch from
any ref other than `main`, so every dispatched release runs `main`'s current SemVer gate. Tag
pushes no longer trigger the workflow at all (the `push: tags: v*` trigger was removed in #222),
closing the prior gap where a tag deliberately cut at a pre-gate commit could run an ungated
workflow. The `publish` environment's deployment-branch policy is pinned to `main` only: the apply
step prunes any non-`main` policy and asserts the allowlist contains exactly one entry (`main`), so
no other ref can deploy.

The universal backstop for every publish path is therefore the `publish` environment's required
reviewer (above): no npm version reaches the registry without a human approving that specific
deployment. The SemVer gate is automated defense-in-depth layered on top of that human gate, not a
replacement for it. The autonomous-merge safety invariant — _nothing publishes without human
approval_ — holds on every path. The old-tag SemVer gap is now closed structurally:
`main`-dispatch is the only publish trigger (#222), so both the human required-reviewer gate and
the automated SemVer gate run on `main`'s current workflow for every release.

The release path also enforces API-surface stability — the relocation of the former per-PR SemVer
check. `release.yml` runs a `SemVer gate (release)` job (pinned `cargo-semver-checks`) that
`publish-all` depends on, so a semver-incoherent tag cannot publish. `scripts/publish.sh` runs the
same check before any `cargo publish` to crates.io, covering the library-publish path that does not
flow through `release.yml`. cargo-semver-checks validates that the public-API diff is consistent
with the version bump; at release the version has bumped, so a breaking change shipped under a
non-breaking bump fails the gate. This is the boundary where the check is both meaningful and
green-able, unlike the per-PR position (section 1). `khive-quant` (no crates.io baseline yet) is
excluded until its first publish.

### 4. Gate-defining files and structural defense

A PR that modifies the gate wall's own scripts or workflows could, if that PR merges
autonomously, weaken the criteria that future PRs must satisfy. This is a self-approval problem
that branch protection cannot eliminate in the general case, because a `pull_request` workflow
runs the head's own files.

The structural mitigation is depth of the wall: no single script or workflow file defines the
full gate. `CI (ubuntu-latest)` and `CI (macos-latest)` run `scripts/ci.sh`; modifying that
script changes what those jobs check. But `Supply-chain (cargo-deny)`, `Secret scan
(gitleaks)`, `Dependency review`, `Doc build (-D warnings)`, and `Coverage
ratchet` run through independent mechanisms. A PR that weakens `ci.sh` does not disable the
other contexts. The only PR that could weaken the full wall simultaneously is one that modifies
every gate in a single diff, which is both impractical and conspicuous.

A second layer of structural defense is the release gate. A gate change that slips through
autonomously is bounded in blast radius by the trunk; it does not automatically reach published
artifacts. The human who approves the next release has an opportunity to observe any anomaly in
`main`'s history before approving publication.

This ADR does not restore CODEOWNERS-based holds on gate-defining paths. Doing so would
re-introduce a per-PR human gate for the paths most likely to be touched by infrastructure work,
which contradicts the full-autonomy decision. The structural argument above is the substitute.

### 5. Non-automatable floor (reversibility test)

These properties are enforced at all times and are never relaxed:

- The secret scan and the data-leak guard are hard-required. A leaked secret or a committed
  dataset is not correctable by a revert because the history persists. The gitleaks scan covers
  full commit history, not only the diff.
- Branch protection forbids force-push and history rewrite on `main`.
- Squash-merge keeps each merge revertable in isolation with a single `git revert`. This is not
  a guarantee that any revert is safe after dependent changes merge: reverting a lower-layer
  change after a dependent change has landed can break the build. The dependency chain is
  foundation to platform to features. Post-merge monitoring and fast revert discipline are the
  operational backstops.
- Publishing stays human-gated. The npm release is held by the `publish` environment's required
  reviewer (section 3); the crates.io library publish is a maintainer-run `make publish` with no
  autonomous CI path. A published version cannot be unpublished after the registry retains it.

## Activation order

The safety property of this design depends on applying the changes in the correct order. Flipping
required approvals to zero before the full gate wall is enforced opens a window in which PRs
merge over a thinner wall. The order is:

1. Land the companion gate PRs (#215 and the new-gates PR) on `main`. These establish the
   `Supply-chain (cargo-deny)`, `No-stub guard`, `Doc build (-D warnings)`,
   `Dependency review`, and `Coverage ratchet` contexts. Until a context's job exists on `main`,
   do not add it to the required set in the ruleset; a required check that never reports
   permanently blocks merge.
2. Create and configure the `publish` GitHub Environment: add it to `release.yml`'s publish
   jobs, add the required reviewer. Validate on a dry-run tag that the environment gate fires
   before any publish step.
3. Update ruleset 17362266: add all gate-wall contexts listed in section 1 as required status
   checks. Keep `required_approving_review_count: 1` and CODEOWNERS intact at this point so the
   wall is enforced before human review is removed.
4. Validate the full wall against a real PR: confirm every required context reports and passes;
   confirm auto-merge fires on a passing PR; confirm the environment gate holds on a test tag.
5. Set `required_approving_review_count: 0` and `require_code_owner_review: false`. Remove or
   archive the CODEOWNERS file. At this point autonomous merge is live.

Step 5 is the last step because review removal is the irreversible action. Every other step
strengthens the safety envelope; step 5 removes the human backstop. The wall must be verified
operational before the backstop is removed.

## Consequences

**Positive.** Every PR merges without waiting on a human, at any PR volume. There is no
maintainer-throughput limit on trunk delivery. The gate wall is broader and more mechanically
complete than a single human reviewer: it catches format, lint, test regression, supply-chain
CVEs, API breakage, documentation rot, coverage regression, secrets, and data leaks
simultaneously. The release gate preserves deliberate human judgment at the point of publication,
where the blast radius of a bad decision is largest.

**Negative and risks.** This model loses something real and should be stated honestly.

Human code review catches a class of defects that deterministic gates cannot: logic errors that
are syntactically and semantically correct Rust, design decisions that contradict unstated
requirements, subtle security vulnerabilities that do not match known patterns, and architectural
drift that accumulates across multiple individually-passing PRs. These defects will reach `main`
and must be caught by the release gate, by post-merge monitoring, or by a revert after
discovery.

The gate wall's breadth reduces but does not eliminate this class. A broad wall of independent
checks is harder to circumvent accidentally than a single CI job, and the release gate provides
a periodic human inspection point, but the gap between a trunk commit and a release review can
span many PRs. Any one of them could carry an undetected logic error.

Mitigations: the squash-merge strategy keeps `main` revert-friendly one commit at a time; the
release gate imposes a mandatory human checkpoint before artifacts reach users; post-merge
regression gates (`ann-ci-gate` re-runs on every push to `main`) provide defense in depth on
the performance-sensitive paths; the coverage ratchet detects regression in test coverage as a
proxy for unverified behavior.

This is a deliberate trade: throughput and automation depth over per-PR human eyeballs. The
trade is appropriate when the gate wall is thick, the release cadence gives the human gate
meaningful sampling frequency, and the development model generates high PR volume from
well-specified automation.

## Alternatives considered

- **Two-lane model with CODEOWNERS hold (prior ADR-066 draft).** Routes architecturally-
  significant paths to mandatory human review; Lane-1 merges autonomously. Retains a human in
  the critical path for the highest-risk PRs, which provides stronger coverage of the logic-
  error class. Rejected in favor of the fully-autonomous model because it recreates a throughput
  limit (the human reviewer) for exactly the changes where automation intensity is highest, and
  because the gate wall plus release gate provide the same class of protection at a lower
  operational cost.
- **Human reviews every PR.** Correct at low volume; does not scale to agent-generated PR
  volume. The maintainer becomes the bottleneck.
- **Autonomous merge without a release gate.** Removes the human checkpoint entirely. Rejected:
  the release gate costs nothing in the normal flow (a human already decides when to tag), and
  it provides a mandatory sampling point that is proportional to publication risk.
- **LLM-based automated reviewer as a required check.** The prior draft included an LLM
  reviewer that classifies each diff and blocks on design violations or escalates to Lane 2. In
  the fully-autonomous model, a required LLM reviewer reintroduces a fail-closed outage risk
  (an API outage holds every open PR) and a false-positive risk (an over-cautious reviewer
  becomes a throughput limit). The deterministic gate wall is preferred because it is
  reproducible, has no running cost per PR, and fails deterministically. An LLM reviewer may be
  added in advisory mode (non-required) to provide signal to the release reviewer; it is
  explicitly not a gate in this model.
- **No auto-merge, faster human review.** Does not address the volume problem; the bottleneck
  remains.
