# ADR-066: Two-Lane PR Merge Pipeline

**Status**: Proposed
**Date**: 2026-06-22
**Relates to**: CI workflow (`.github/workflows/ci.yml`), `scripts/ci.sh`, `CODEOWNERS`, repository branch protection

---

## Context

khive receives a high and growing volume of pull requests, many generated by automated agents
working from issues. Hand-reviewing every PR does not scale: it makes the maintainer the
bottleneck and delays conforming changes that carry no architectural risk. At the same time, a
fraction of PRs touch the parts of the system where a wrong merge is expensive or hard to
reverse: the ADR contract, the database schema, the closed taxonomies, the public MCP surface,
the authorization seam, the dependency set, and the CI gates themselves.

The repository already enforces a substantial deterministic floor in CI: format, clippy under
`-D warnings`, the full workspace test suite, a no-default-features build, a release build, SQL
lint, contract/smoke/deno tests, a JSON data-leak guard, a gitleaks secret scan, a docs-format
check, and a marketplace-example validator. What it lacks is (a) a few additional deterministic
gates, (b) an automated semantic review of each diff, and (c) a defined policy for which PRs may
merge without human review and which must not.

## Decision

Adopt a two-lane merge pipeline.

**Lane 1 — conforming changes.** A PR that passes the full deterministic gate set and the
automated reviewer, and that does not touch any architecturally-significant path, is merged
automatically (squash) and its branch deleted. No human review is required.

**Lane 2 — architecturally-significant changes.** A PR that touches an architecturally-significant
path, or that the automated reviewer flags as ambiguous or risky, is held for maintainer review
before merge. The hold is enforced natively (CODEOWNERS), not by an opt-out bot — see §4.

A non-automatable floor (§5) is never delegated to automation, in either lane.

## Mechanism

### 1. Deterministic gate set (required status checks)

Branch protection on `main` requires exactly these status contexts. The strings are matched by
name, so they must be exact:

- `CI (ubuntu-latest)` and `CI (macos-latest)` — the matrix job running `scripts/ci.sh`: fmt,
  clippy `-D warnings`, workspace tests, no-default-features check, release build, SQL lint, the
  no-stub guard, and contract/smoke/deno tests. Both legs of the matrix are required.
- `Supply-chain (cargo-deny)` — license, bans, and sources policy (see `crates/deny.toml`).
- `JSON/JSONL data-leak guard`.
- `Secret scan (gitleaks)`.
- `Docs lint`.
- `Marketplace example validator`.

This is the gate set **after** the companion CI PR (the cargo-deny job and the no-stub guard)
lands on `main`. A required status check that never reports blocks merge (this is GitHub's
fail-safe default), so branch protection must not require the `Supply-chain (cargo-deny)` context
or the no-stub-bearing `CI` context until that PR is merged — otherwise every PR wedges on a
never-reported check. Rollout step 1 enforces this ordering.

The no-stub guard (`scripts/check-no-stubs.sh`) flags the `todo!`, `unimplemented!`, and `dbg!`
macros in shipping source (`crates/*/src/**`, excluding tests/benches/examples). It is a
debug-and-stub-macro screen, not a proof of completeness; placeholder patterns that are not those
three macros are not caught, and they sit below the reversibility floor (§5) in any case.

Conditionally-run workflows are deliberately **not** in the required set. Both regression gates
are `pull_request` checks gated on a path filter, so on a PR that does not touch their paths the
whole workflow is skipped and its check context never reports — it stays permanently "expected,"
which a required check treats as pending. A required check that never reports blocks merge, so
requiring either of these would wedge every PR that does not touch those paths. (This is the
single most important auto-merge configuration detail: requiring a conditionally-run check
silently disables auto-merge for the majority of PRs.)

- `Pipeline Regression Gate` (`bench-pipeline.yml`, context `Pipeline Regression Gate`) runs on
  PRs that touch the retrieval/storage/memory stack (`khive-retrieval`, `khive-fusion`,
  `khive-bm25`, `khive-vamana`, `khive-hnsw`, `khive-score`, `khive-db`, `khive-runtime`,
  `khive-mcp`, `kkernel`, `khive-pack-memory`). Its `pipeline-gate` job is blocking.
- `Vamana ANN Regression Gate` (`bench-1m.yml`) runs its blocking `ann-ci-gate` job (context
  `ANN structural regression gate (synthetic, hermetic)`) on both push to `main` and on PRs that
  touch `khive-vamana` and the bench scripts. Its second job, the real SIFT-1M scale-proof, is
  `workflow_dispatch`-only. So this workflow _is_ a PR check, contrary to a naive "push-only"
  reading; it is excluded from the required set for the never-reports-when-skipped reason above,
  not because it is post-merge.

Excluding these from the required set leaves a gap that branch protection cannot close on its own:
GitHub required-status-contexts are global, with no native "required only when path X changes," so
a conditional regression gate cannot be both non-wedging (not required) and auto-merge-blocking. A
non-required check, even a failing one, does not hold auto-merge. The gap is closed structurally:
the performance-contract engine crates these gates guard are routed to Lane 2 (§3), so a PR that
could regress recall or latency gets a human reviewer who is the backstop for the gate that cannot
be globally required, while the bench still runs and its result informs that reviewer and the
automated reviewer (§2). The `ann-ci-gate` job additionally re-runs post-merge on every push to
`main`, defense in depth for the Vamana path; `bench-pipeline.yml` has no post-merge trigger, so
its only backstops are the Lane-2 review and the automated reviewer. Changing a bench threshold is
itself a gate-defining change, held by the `.github/**`, `scripts/**`, and `perf/targets.toml`
routing in §3.

### 2. Automated reviewer gate

A GitHub Action runs an LLM reviewer on each PR diff, with the ADR set and the closed-taxonomy
definitions in context. The reviewer returns a structured verdict and sets a status check. It
blocks on:

- security or correctness defects,
- scope creep beyond the PR's stated purpose,
- violations of an ADR or a closed taxonomy,
- removal or weakening of a test or a gate.

It also escalates to Lane 2 (rather than blocking outright) any Lane-1 PR whose conditional
regression-gate check (the bench gates in §1) is red or failing, since those gates cannot be
globally required and a perf regression on an unlisted path would otherwise ride Lane 1.

The reviewer also classifies the PR's lane and escalates anything it judges ambiguous to Lane 2.
It runs on GitHub's runners using an API key held in repository secrets, so it scales without a
human in the loop.

The reviewer must fail safe operationally, and the safe direction depends on its mode. While
advisory (pre-promotion), an outage is harmless: the reviewer is non-blocking, deterministic gates
carry the PR, and it simply does not weigh in. Once promoted to a required check, the fail-safe
direction inverts: if the Action errors, its API key is exhausted or rotated, or the provider is
unavailable (all plausible at high PR volume), the reviewer emits a non-success result rather than
hang. Because it is required, that holds auto-merge on every open PR until the reviewer recovers or
the maintainer clears PRs via the branch-protection bypass. This is deliberately fail-closed: a
reviewer outage pauses auto-merge rather than letting PRs through unreviewed — including a Lane-1
PR carrying a red conditional bench check (§1), whose only escalation backstop is the reviewer
itself, so the escalation and the outage share fate and the failing required check is what holds
the PR. A short outage costs throughput (auto-merge pauses), not safety. The reviewer is introduced
in advisory mode and promoted to required only after a calibration window confirms its verdict
quality and its cost ceiling at the repository's PR volume.

### 3. Lane classification

A PR is Lane 2 (merge held, maintainer review required) when its diff touches any of these paths.
They fall in two groups: changes to the system's contract, and changes to the gates themselves.

Contract paths:

- `docs/adr/**` — the ADR contract.
- `crates/khive-db/sql/**` and `crates/khive-db/src/migrations.rs` — database schema and
  migrations.
- `crates/khive-pack-kg/src/vocab.rs` — the entity and note kind taxonomies.
- `crates/khive-types/src/edge.rs` — the `EdgeRelation` closed enum.
- `crates/khive-mcp/src/**` and `crates/khive-request/src/**` — the public MCP surface and its
  request DSL.
- `crates/khive-gate/**` and `crates/khive-gate-rego/**` — the authorization seam, the one trust
  boundary (ADR-018).
- `crates/Cargo.toml` — the workspace member and exclude lists (adding or dropping a crate from
  the build changes what is compiled and tested).
- any other `Cargo.toml` or `Cargo.lock` change that adds, removes, or bumps a dependency.
- `crates/khive-vamana/**`, `crates/khive-hnsw/**`, `crates/khive-retrieval/**`,
  `crates/khive-fusion/**`, `crates/khive-bm25/**`, `crates/khive-score/**` — the ANN, retrieval,
  fusion, and scoring engines, whose performance contract (recall, latency, beam growth) is
  guarded by the conditional regression gates. Those gates cannot be globally required without
  wedging unrelated PRs (§1), so a change here is reviewed by a human as the backstop.
  (`khive-mcp` source is already Lane 2 via the MCP-surface entry above. The remaining crates on
  the bench path filters stay Lane 1 — too high-churn to hold every PR for review: `khive-runtime`,
  `kkernel`, `khive-pack-memory`, and the non-schema internals of `khive-db` (its `sql/**` and
  `migrations.rs` are Lane 2, but its storage internals — for example the `VectorStore`
  implementation in `src/stores/` — are not). These rely on the automated reviewer, which escalates
  to Lane 2 on a failing bench check (§2); the fail-closed outage behavior in §2 means a reviewer
  outage holds rather than releases them.)

Gate-defining paths (a change here changes what "green" means, so it must not be self-approvable):

- `.github/**` — the workflows that run the checks, the `CODEOWNERS` file, and the reviewer
  Action.
- `scripts/**` — the gate scripts (`ci.sh`, `check-no-stubs.sh`, `lint-sql.sh`, the data-leak
  guard, `publish.sh`).
- `crates/deny.toml` — the supply-chain policy.
- `perf/targets.toml` — the regression-gate thresholds (recall floors, beam-growth ceilings). A
  change here loosens or tightens what the bench gates assert, so it redefines "green" the way a
  script change does; the bench would otherwise pass under a loosened floor without a human seeing
  it.

Everything else is Lane 1. The automated reviewer may promote a Lane-1 PR to Lane 2 on judgment.

### 4. Auto-merge mechanics (fail-safe Lane-2 hold)

The Lane-2 hold must be fail-safe: a misclassification or a bot defect must never let an
architectural change merge without review. GitHub's native auto-merge, once permitted, merges any
PR whose required conditions are met, so "simply not enabling auto-merge on a Lane-2 PR" is
fail-dangerous — it relies on a bot to _not act_, and a single bot defect merges the exact change
the design means to hold. The hold is therefore enforced natively:

- A `CODEOWNERS` file maps every Lane-2 path (§3) to the maintainer.
- Branch protection sets "Require a pull request before merging" with **0 required approvals** and
  "Require review from Code Owners" enabled. A PR that touches a CODEOWNERS-matched path cannot
  merge until a code owner approves; a PR that touches no matched path needs no approval. (The
  GitHub docs do not explicitly state that 0 required approvals composes with code-owner
  enforcement, so rollout step 2 validates it on a throwaway PR before the pipeline depends on
  it.)
- The proposer and the code owner must be distinct identities. GitHub forbids approving one's own
  PR, so if the sole code owner authors a Lane-2 PR the code-owner requirement is unsatisfiable and
  the PR is held indefinitely. Agent-generated PRs (the high-volume majority) must therefore be
  authored by a bot identity, with the maintainer as the code owner who clears Lane 2; the
  maintainer's own occasional Lane-2 PRs merge via the admin bypass. Every PR today is authored by
  the maintainer's account, so giving agent pushes a distinct identity is a precondition for the
  hold to function, not an optional refinement — validated alongside rollout step 2.
- Auto-merge is permitted repository-wide (`allow_auto_merge`); squash is the merge method (one
  revertable commit per merge); delete-branch-on-merge is enabled.
- A lightweight workflow enables auto-merge (`gh pr merge --auto --squash`) on every PR when it
  becomes ready for review (the `ready_for_review` event — draft PRs cannot carry auto-merge).
  Enabling auto-merge on a Lane-2 PR is safe: GitHub holds the merge until the required code-owner
  review arrives. The bot is belt-and-suspenders; the native code-owner requirement is the floor.
- Branch protection also enables "Require branches to be up to date before merging" and "Dismiss
  stale pull request approvals when new commits are pushed," so a green result or an approval
  cannot be carried stale across a base advance or a post-review force-push.

A same-repo `pull_request` workflow runs the PR head's own workflow and script files, so a PR can
weaken the gate that runs on itself (for example by editing `scripts/ci.sh` to skip a step while
keeping the job name green). Routing `.github/**` and `scripts/**` through CODEOWNERS (§3) is what
prevents a PR from self-approving a weakened gate; there is no pure-CI defense against a check
redefining its own pass criteria.

One GitHub constraint to note: the default `GITHUB_TOKEN` can enable auto-merge, but events it
triggers do not start new workflow runs. The auto-merge-enabling workflow therefore only enables
auto-merge on PRs whose checks were already triggered by the human or agent push that opened them;
it does not itself push commits that would need CI.

### 5. Non-automatable floor (reversibility test)

These are never delegated to automation, in any lane:

- The secret scan and the data-leak guard are hard-required. A leaked secret or a committed
  dataset is not made safe by a revert; the history persists. (The gitleaks scan runs over full
  history, not just the diff.)
- Branch protection forbids force-push and history rewrite on `main`.
- Squash-merge keeps each merge revertable **in isolation** with a single `git revert`. This is
  not a guarantee that any merge can be reverted safely after the fact: reverting a lower-layer
  change once a dependent change has merged can break the build, because the dependency chain runs
  foundation → platform → features. Schema and dependency changes are exactly the ones that need
  human sequencing, which is why §3 routes them to Lane 2.
- Publishing to crates.io stays a manual, human-gated step. A merge is not a release, and a
  published version cannot be unpublished.

## Rollout

1. Land the new deterministic gates (cargo-deny job, no-stub guard) on `main` via the companion CI
   PR. Until this lands, do not require its contexts in branch protection.
2. Add the `CODEOWNERS` file mapping the Lane-2 paths (§3) to the maintainer. On a throwaway PR,
   verify that 0 required approvals plus "Require review from Code Owners" actually holds a
   CODEOWNERS-matched PR while letting an unmatched PR through. Author that PR from the intended
   bot identity, so the test also confirms the maintainer can clear the hold — a
   maintainer-authored PR cannot be self-approved, so agent pushes need a distinct identity (§4).
3. Enable branch protection: require the deterministic gate set (the exact contexts in §1),
   require code-owner review, require branches up to date, dismiss stale approvals, forbid
   force-push. Permit auto-merge, set squash as the merge method, enable delete-branch-on-merge,
   and add the auto-merge-enabling workflow.
4. Add the automated reviewer in advisory mode; run it on live PRs for a calibration window, tuned
   conservatively (biased toward escalating to Lane 2), with the operational fallback in §2.
5. Promote the reviewer to a required check once its verdicts and cost are calibrated.

## Consequences

**Positive.** Conforming PRs merge without waiting on a human. Maintainer attention is spent only
where it changes the outcome: the architecturally-significant minority, which is held natively by
CODEOWNERS rather than by a bot's restraint. The deterministic floor plus the reviewer catch more
classes of defect before merge than the prior CI did. The gate-defining files cannot be changed by
a self-approving PR. The reversibility floor holds for the failures a revert cannot undo.

**Negative.** The automated reviewer has a running cost and can produce false verdicts; the
advisory window, conservative tuning, and the maintainer bypass reduce but do not remove this. An
API key in repository secrets is a new secret to manage and rotate. The Lane-2 path list is the
authoritative boundary — a genuinely architectural change in an unlisted path would ride Lane 1,
so the list is maintained as the system grows and the reviewer's judgment-based escalation is the
backstop. CODEOWNERS concentrates the human gate on one maintainer, who becomes the throughput
limit for Lane-2 PRs. One residual is time-boxed to the reviewer's advisory window: until the
reviewer is required and fail-closed, it is not yet a gate, so a perf regression on a Lane-1
bench-path crate can merge, and because `bench-pipeline.yml` has no post-merge trigger it is not
caught post-merge there either — the bench still surfaces it on the PR for a human who looks, and
the window is closed by promoting the reviewer.

## Alternatives considered

- **Human reviews every PR.** The status quo. Correct at low volume; it does not scale to
  agent-generated PR volume and makes the maintainer the bottleneck.
- **Auto-merge on deterministic CI only, no reviewer.** Simple, but deterministic checks do not
  catch design defects, scope creep, or ADR violations — the failures a human reviewer caught.
- **Lane-2 hold by a classifier bot that withholds auto-merge.** Rejected as fail-dangerous: a
  single bot defect auto-merges an architectural change. CODEOWNERS makes the hold native and
  fail-safe.
- **No auto-merge, faster human review.** Does not address the volume problem; the bottleneck
  remains.
